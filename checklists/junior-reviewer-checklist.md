# Junior Reviewer チェックリスト

**エス・エー・エス株式会社**  
*新人・初級レビュアー向けチェックリスト*

## 📚 はじめに

このチェックリストは、PRレビューを始めたばかりの方や、レビュースキルを向上させたい方向けに作成されています。基本的かつ重要な項目に焦点を当て、段階的にレビュースキルを向上させることを目的としています。

## ✅ 必須チェック項目

### 1. 基本的な品質確認
```markdown
## コーディング規約
- [ ] 変数名・関数名が規約に従っているか
  - キャメルケース、スネークケースの統一
  - 意味のある名前になっているか
- [ ] インデントが統一されているか
  - タブまたはスペースの統一
  - インデントレベルの適切性
- [ ] 不要なコメントアウトが残っていないか
- [ ] console.logやデバッグコードが残っていないか
```

### 2. 明らかなバグの確認
```markdown
## バグチェック
- [ ] 明らかな構文エラーがないか
- [ ] 変数の未定義参照がないか
- [ ] 無限ループの可能性がないか
- [ ] 配列の範囲外アクセスがないか
- [ ] NULL/undefinedの不適切な参照がないか
```

### 3. テストの確認
```markdown
## テスト実行
- [ ] 既存のテストが通っているか
- [ ] 新しい機能にテストが追加されているか
- [ ] テストの内容が適切か（何をテストしているか理解できるか）
- [ ] テストケースの名前が分かりやすいか
```

### 4. PR説明との整合性
```markdown
## PR内容確認
- [ ] PR説明に書かれた変更が実装されているか
- [ ] 変更内容が理解できるか
- [ ] 不要な変更が含まれていないか
- [ ] コミットメッセージが適切か
```

### 5. セキュリティ基本確認
```markdown
## セキュリティ
- [ ] パスワードやAPIキーがハードコードされていないか
- [ ] 個人情報が含まれていないか
- [ ] SQLクエリに変数が直接埋め込まれていないか
- [ ] 機密情報がログに出力されていないか
```

## 🎯 レビューのポイント

### コードを読む際の着眼点

#### 1. 変数名・関数名をチェック
```javascript
// ❌ 悪い例
function calc(x, y) {
  return x * 1.08 + y;
}

// ✅ 良い例
function calculateTotalWithTax(price, shippingFee) {
  const TAX_RATE = 1.08;
  return price * TAX_RATE + shippingFee;
}
```

#### 2. エラーハンドリングをチェック
```javascript
// ❌ 悪い例
function getUserData(userId) {
  const user = database.find(userId);
  return user.name; // userがnullの場合エラー
}

// ✅ 良い例
function getUserData(userId) {
  const user = database.find(userId);
  if (!user) {
    throw new Error(`User not found: ${userId}`);
  }
  return user.name;
}
```

#### 3. マジックナンバーをチェック
```javascript
// ❌ 悪い例
if (items.length > 100) {
  // 100は何を意味する？
  showPagination();
}

// ✅ 良い例
const MAX_ITEMS_PER_PAGE = 100;
if (items.length > MAX_ITEMS_PER_PAGE) {
  showPagination();
}
```

## 💡 学習ポイント

### レビューコメントの書き方

#### 良いコメントの例
```markdown
✅ 建設的なコメント：
「この関数は処理が複雑になっているようです。
`calculatePrice()`と`applyDiscount()`に分割すると
可読性が向上すると思います。いかがでしょうか？」

❌ 避けるべきコメント：
「このコードは読みにくいです。」
```

#### 質問の仕方
```markdown
✅ 良い質問：
「このロジックの意図を教えていただけますか？
○○の場合の処理が見当たらないように思えるのですが、
別の箇所で処理されているのでしょうか？」

❌ 避けるべき質問：
「これは何をしているんですか？」
```

## 🚀 スキルアップのために

### 段階的な成長目標

#### Level 1（最初の1ヶ月）
- [ ] 基本的なコーディング規約違反を見つけられる
- [ ] 明らかなバグを指摘できる
- [ ] 建設的なコメントが書ける

#### Level 2（2-3ヶ月）
- [ ] パフォーマンスの問題に気づける
- [ ] テストの品質を評価できる
- [ ] 設計の改善提案ができる

#### Level 3（4-6ヶ月）
- [ ] セキュリティリスクを識別できる
- [ ] アーキテクチャレベルの議論に参加できる
- [ ] 他のレビュアーをサポートできる

## 🔍 よくある見落としポイント

### 初心者が見落としがちな項目
1. **エラーメッセージの内容**
   - ユーザーフレンドリーか
   - デバッグに役立つ情報が含まれているか

2. **ログ出力**
   - 適切なログレベルか
   - 機密情報が含まれていないか

3. **設定値のハードコード**
   - 環境変数や設定ファイルに切り出すべきか

4. **国際化対応**
   - ハードコードされた日本語/英語がないか

5. **タイムゾーン**
   - 日時処理でタイムゾーンが考慮されているか

## 💬 困ったときは

レビューで判断に迷った場合：
1. **先輩に相談**: 不明な点は遠慮なく質問
2. **ガイドライン確認**: 社内ガイドラインを参照
3. **チームで議論**: Slackの#code-reviewチャンネルで相談

## 🎉 レビュアーとしての心構え

- **学ぶ姿勢**: 他の人のコードから学ぶ機会と捉える
- **建設的**: 批判ではなく改善提案を心がける
- **謙虚**: 自分も間違える可能性があることを認識
- **感謝**: 良いコードには積極的に褒める

**Remember**: 完璧なレビューを目指す必要はありません。少しずつ成長していきましょう！
